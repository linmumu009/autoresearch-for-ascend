# v0.4.0 - ms-swift DPO LR Search

## Scope

Started a separate ms-swift learning-rate search for:

- model: Qwen3.6-27B
- method: DPO, not SFT/PPO
- adapter: LoRA rank 8, alpha 32
- sharding: FSDP2 full shard
- hardware: 8 visible Ascend NPUs in `llin-rl-dpo`
- data: `/workspace/llin-rl-dpo/datasets/ops_dpo_512.jsonl`

## Environment Notes

An isolated workspace copy was created first, and a copied container was tested.
The copied image preserved the Python/ms-swift stack, but did not inherit a
working NPU runtime view. To avoid changing the host or unrelated containers,
the actual LR experiments ran in the already verified `llin-rl-dpo` container
with isolated output directories under:

```text
/workspace/llin-rl-dpo/outputs/ms-swift-qwen36-dpo-lrsearch/
```

The runner explicitly sets the known-good Ascend runtime library path before
calling `swift rlhf`; without this, `torch_npu` imported the wrong toolkit path
and failed on `libruntime_common.so`.

## Conclusion

For this short-budget DPO surface, the best LR so far is `2.0e-4`.

`4.0e-4` is still numerically stable, but it starts to push chosen rewards down
while relying more on rejected suppression, and its holdout loss is worse than
`2.0e-4`. The next useful check is a longer `2.0e-4` confirmation run with the
same split, then a small downstream generation comparison.
