# v0.4.1 - 100-step ms-swift DPO Confirmation

## Scope

Confirmed the v0.4.0 learning-rate winner, `2.0e-4`, with a longer 100-step
run using the same Qwen3.6-27B + DPO + LoRA + FSDP2 setup.

## Command Shape

```bash
RUN_ID=lr_2e-4_100step_eval10p_20260704 \
LEARNING_RATE=2e-4 \
MAX_STEPS=100 \
SPLIT_DATASET_RATIO=0.1 \
EVAL_STRATEGY=steps \
EVAL_STEPS=50 \
SAVE_STRATEGY=no \
bash scripts/run_ms_swift_lr_candidate.sh
```

## Results

| Eval Step | Eval Loss | Eval Margin | Eval Chosen | Eval Rejected |
| ---: | ---: | ---: | ---: | ---: |
| 50 | `2e-8` | `19.71` | `2.79` | `-16.91` |
| 100 | `2e-8` | `19.84` | `2.69` | `-17.18` |

Summary:

- train_loss: `0.01312333`
- runtime: `356.3s`
- train_steps_per_second: `0.281`
- memory: about `51.19` GiB per NPU

## Conclusion

`2.0e-4` remains the recommended learning rate for this setup.

The DPO validation surface is already saturated by 50 steps. The next useful
experiment should compare generation outputs or preference win-rate using saved
adapters, because lower DPO loss alone is no longer very informative.
