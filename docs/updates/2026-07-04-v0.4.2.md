# v0.4.2 - Adapter Save and Generation Smoke

## Scope

Saved a real LoRA adapter for the recommended `2.0e-4` DPO learning rate and
tested whether it changes short-form generation compared with the base model.

## Runner Change

`framework_adapters/ms_swift_dpo/run_lr_candidate.sh` now exposes:

- `SAVE_STEPS`
- `SAVE_TOTAL_LIMIT`

This lets the same LR-search runner produce a usable checkpoint without editing
the script.

## Generation Smoke

The saved adapter was compared against the base model through
`scripts/run_ms_swift_base_adapter_compare.sh`.

| Prompt Set | Decoding | Rows | Changed Outputs |
| --- | --- | ---: | ---: |
| `datasets/fixed_eval_prompts.jsonl` | greedy, `temperature=0` | 2 | 0 |
| `datasets/ops_eval_prompts_16.jsonl` | greedy, `temperature=0` | 16 | 0 |
| `datasets/ops_eval_prompts_16.jsonl` | `temperature=0.7` | 16 | 0 |

Inference logs confirmed the adapter path was used and ms-swift instantiated
`PeftModelForCausalLM`.

## Conclusion

The DPO objective is moving the preference/logprob surface, but this checkpoint
does not change short prompt generation in the current smoke tests. The next
step should be an explicit preference/logprob scorer or a larger generation
benchmark rather than relying on tiny fixed prompt text equality.
