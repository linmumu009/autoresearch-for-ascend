# v0.4.3 - DPO Pair Logprob Scorer

## Scope

Added an explicit chosen/rejected logprob scorer for the ms-swift DPO adapter.
This tests the preference surface directly, instead of relying on whether short
generation strings change.

## Method

For each DPO row:

1. build the chat prompt from system/user messages,
2. score the chosen response tokens,
3. score the rejected response tokens,
4. compute `margin = chosen_mean_logprob - rejected_mean_logprob`.

The scorer runs base and adapter separately so the 27B model is not loaded twice
inside one process.

## Results

Dataset: first 64 rows of `/workspace/llin-rl-dpo/datasets/ops_dpo_512.jsonl`.

| Model | Rows | Mean Chosen Logprob | Mean Rejected Logprob | Mean Margin | Win Rate |
| --- | ---: | ---: | ---: | ---: | ---: |
| Base Qwen3.6-27B | 64 | `-4.0987` | `-4.7941` | `0.6954` | `81.25%` |
| `2.0e-4` LoRA checkpoint-50 | 64 | `-2.7799` | `-9.7350` | `6.9551` | `100%` |

Per-row margin deltas:

- improved rows: `64/64`
- mean delta: `+6.2597`
- median delta: `+6.2749`
- minimum delta: `+4.1543`
- maximum delta: `+8.6847`

## Conclusion

The saved `2.0e-4` LoRA adapter clearly changes the DPO preference/logprob
surface even though v0.4.2 short-prompt generation text stayed identical.

This is the right interpretation: the adapter can strongly re-rank chosen over
rejected continuations without necessarily changing the top-1 generated string
for short prompts. The next step should score held-out pairs or construct a
larger generation benchmark with prompts where the desired answer differs more
from the base model.
